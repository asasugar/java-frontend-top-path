# Job 16 — 全栈闭环验收

承接 [job-15.md](job-15.md)：用**一条命令或明确三步内**起全套，**前端真实调后端**，完成转型路线入门段的**收口**。

## 目标

- **Compose 或脚本**：`app + db + redis (+ broker 可选)` 一次起。
- **前端项目**（你熟悉的 Vue/React 等）：**baseURL** 指向后端，完成 **登录（若已实现）→ 列表/创建** 一条用户路径。
- **OpenAPI**：前端用生成客户端或手写类型，**字段与后端一致**。
- **新人可读**：根或子项目 **README** 写清：前置条件、启动命令、默认账号（若有）、常见端口。

## 阅读 × 前端类比

| 概念 | 前端可类比 |
|------|------------|
| 全栈闭环 | **E2E 环境可复现**：像 **Vercel Preview + 后端 staging** 一次配齐，而不是「我本地 magic」。 |
| 契约驱动 | **Orval 生成 hooks**：改 OpenAPI 再生成，**类型错误在编译期爆**，减少联调扯皮。 |

## 任务（验收清单）

- [ ] 克隆仓库的队友（或未来的你）按 README **30 分钟内**跑通 **端到端**。
- [ ] 浏览器 **Network** 可见 **带鉴权** 的请求 **2xx**，错误时响应体符合 Job 05 约定。
- [ ] **CORS**、**Cookie / Bearer** 方案与部署域一致（无「本地行、上线挂」）。
- [ ] **docker compose up** 或 **k8s + ingress** 任选一种作为「标准演示环境」写死文档。
- [ ] 笔记：整条路线 **最大收获 + 下一步想深入的方向**（如 MyBatis、网关、GRPC）。

## 排错

- **混用 localhost 与容器名**：前端在浏览器跑，只能访问 **映射到宿主机的端口**，不能写 **Docker service 名**。
- **HTTPS / SameSite**：生产 Cookie 策略与 **Secure** 标记要提前设计。
