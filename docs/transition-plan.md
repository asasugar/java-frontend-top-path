# 转型计划与概念扫盲

面向前端背景的 Java 全栈转型：**可兼职执行**，以下为约 **12～16 个阶段** 的粗粒度路线（节奏可按月自定）；入门与语言巩固见 [jobs.md](jobs.md)（[job-01.md](job-01.md)、[job-02.md](job-02.md)）。

## 一、转型计划（按阶段）

| 阶段 | 主题 | 目标产出 |
|------|------|----------|
| 1～2 | Java 语言与 JVM 初印象 | 独立小程序（读写文件、HTTP）；字节码、classpath、jar；集合与 Stream、线程池入门。**任务**：[job-01.md](job-01.md)、[job-02.md](job-02.md) |
| 3～4 | 构建工具与模块化 | 用 Maven 或 Gradle 建立多模块结构；依赖范围（compile/test/runtime）能选对 |
| 5～7 | Spring Boot 与 Web | 可运行的 REST 服务；校验、分页、统一错误体；OpenAPI 可导出给前端 |
| 8～9 | 数据层 | 接 PostgreSQL/MySQL；JPA 实体 + 迁移脚本（Flyway/Liquibase）；理解事务边界 |
| 10～11 | 缓存与安全 | Redis 缓存典型场景；Spring Security 登录态或 JWT 与前端联调通 |
| 12～13 | 消息与 resilient | 一种消息队列入门；超时、重试、幂等在接口层有意识 |
| 14～16 | 交付与可观测 | Docker 镜像可运行；健康检查、基础指标或日志规范；一次「从前端到部署」闭环 |

**贯穿整段的要求**：每完成一阶段（或每隔一段时间）把本仓库或 side project 推到 Git，并在 issue / 笔记里写一句「最大卡点与结论」。

**与前端工作的衔接**：前半段可把「公司/个人前端项目」的 Mock 换成你本地 Spring 接口，强迫自己练契约与 CORS。

## 二、概念扫盲（从前端类比到后端）

### 1. 运行时与工程

- **JVM**：类似把 JS 丢进 **V8** 里跑；Java 源码先编成 **字节码**，再由 JVM 执行（可 JIT 优化）。你不直接「部署源码」，而是部署 **jar/war** 或容器镜像里的进程。
- **JAR**：像一个 **打过包的依赖包或可执行包**；Spring Boot 常见 **fat jar**（依赖都打进去），类似「单文件包含 node_modules 的可运行产物」（概念类比，实现不同）。
- **Maven / Gradle**：角色接近 **npm + 构建脚本**：拉依赖、编译、测试、打包。`pom.xml` / `build.gradle.kts` 对应 `package.json` 的依赖与脚本，但 Java 侧版本与 **传递依赖** 更要上心。
- **Spring Boot**：不是「语言」，是 **约定优于配置** 的应用框架 + 启动器集合，帮你拼好 Web、配置、指标等骨架；类似在 Node 里选 **Nest** 那种「带模块与注入」的框架，但生态偏企业级。

### 2. 架构与「状态放哪」

- **IoC / DI（控制反转与依赖注入）**：不在类里 `new` 所有协作对象，由容器 **注入** 依赖。类比：前端里不写死具体实现，而用 **参数/Provider 注入**；Spring 用构造器注入最推荐。
- **Bean**：交给 Spring 容器管理的对象（有生命周期）。类似框架里 **单例服务** 注册表里的实例，但作用域可以是 singleton / prototype 等。
- **无前端式全局 Store**：服务端以 **请求为边界**；跨请求状态进 **数据库、缓存、消息**，不进「进程内全局变量」。

### 3. Web 与 HTTP（你已熟，补服务端语义）

- **REST + OpenAPI**：和你写 **fetch/axios + 类型** 一样，契约先行；springdoc 生成 OpenAPI 后，前端可继续用 **Orval / openapi-typescript**。
- **Controller → Service → Repository**：常见三层。类比：**路由处理函数** 薄一点，**业务** 与 **数据访问** 分开，方便测与换存储。
- **DTO 与 Entity**：DTO 给 API 边界用；Entity 常贴近表结构。不要所有表字段都暴露给前端，类似你不把 ORM 模型直接当 API 类型。

### 4. 数据与一致性

- **事务（Transaction）**：多条写要么一起成功要么一起回滚。**隔离级别**（读未提交、读已提交、可重复读、串行化）决定「并发下读到什么」；前端较少接触，但和 **重复下单、库存超卖** 直接相关。
- **传播行为（Propagation）**：一个事务方法调另一个时，是 **加入当前事务** 还是 **新开事务**；错误用法会导致「以为回滚了其实没回滚」。
- **ORM（如 JPA）**：对象与表的映射。注意 **N+1 查询**（像列表里每张脸都去单独请求一次详情）；用 **fetch join / 分批 / DTO 投影** 优化。
- **连接池**：用数据库像用 **HTTP keep-alive 连接池**；池子太小排队，太大占库连接；与 **HikariCP** 等配置相关。

### 5. 缓存（Redis）

- **穿透 / 击穿 / 雪崩**：热点不存在 key、热点过期、大量 key 同时过期导致 DB 被打穿。前端只消费结果，服务端要设计 **空值缓存、互斥重建、随机 TTL** 等。
- **缓存与 DB 一致性**：没有银弹；常见是 **先写库再删缓存** 或 **延迟双删**，需接受短暂不一致并结合业务容忍度。

### 6. 消息队列（Kafka / RabbitMQ）

- **异步解耦**：下单后发消息，库存、通知慢慢消费。类比前端 **事件总线**，但是 **持久化、可重试、可被多消费者读**。
- **幂等**：同一消息可能被投递多次，消费者用 **业务幂等键** 或 **去重表** 保证「执行多次等于一次」。

### 7. 安全

- **Session vs JWT**：Session 像 **服务端记一本账**，客户端只带 cookie id；JWT 像 **自带声明的票**，服务端（或网关）校验签名。SPA 常用 JWT + refresh 或与 BFF cookie 组合。
- **Spring Security**：过滤器链 + 鉴权规则；类似 **网关中间件 + RBAC**，要习惯 **线程内 SecurityContext** 里取当前用户。

### 8. 并发（转型必过门槛）

- **线程**：JVM 里并发执行单元。**虚线程（Virtual Thread）**：更轻量，适合大量阻塞 I/O；不要把它当成「万能，CPU 密集也算线程」。
- **`volatile`、锁、`j.u.concurrent`**：与 **可见性、有序性** 有关；前端 **单线程事件循环** 较少碰这些，但后端写共享可变状态时必须正视。

### 9. 可观测与交付

- **日志 / 指标 / 链路**：对应你前端 **Sentry + console**，生产侧要 **结构化日志**、**RED/USE 类指标**、**分布式 trace id**。
- **健康检查**：容器与编排用 **/actuator/health** 判断进程是否该摘流量，不仅是「进程还在」。

## 三、你何时算「阶段过关」

- **阶段 7**：不开前端也能用 curl/Bruno 把 CRUD 与错误场景跑通，且有一份 OpenAPI。
- **阶段 11**：带登录态的前端能调通主要接口，列表页无明显 N+1。
- **阶段 16**：新人照着 README 能在本地 **一条命令**（或明确三步内）起服务 + DB + 缓存（可用 compose），并能在监控或日志里定位一次失败请求。

可按自身基础跳过已掌握的阶段，但 *事务、并发、安全、幂等* 四条不建议跳过。
